# 测试需求分析 Skill

## 定位

将移动端 App / 小程序 / H5 需求（从几句话到完整 PRD）转化为可用于评审和执行的测试资产，支持两阶段独立使用。

---

## 启动逻辑

收到需求后，先判断用户处于哪个阶段，然后执行对应的流程：

- **需求刚到、尚未澄清、要参加评审** → 执行阶段一
- **需求已澄清、要写测试点/用例** → 执行阶段二（若无阶段一产出，先做阶段一再做阶段二）
- **用户明确说"两阶段都要"** → 顺序执行

如果无法判断，简单问一句：「需求处于评审阶段，还是已经可以开始写测试点了？」

---

## 阶段一：评审期分析（快速输出，不写步骤）

### Step 1 — 被测对象与范围概要

用一张结构化的概要说清楚测试边界，让所有参与方（产品、开发、测试）对齐范围：

```
被测对象：[模块/功能名称，一句话概括其目标]
平台：[App iOS / App Android / 微信小程序 / H5-微信内 / H5-独立浏览器 ...]

✅ 测试范围（IN）：
- [包含的功能点1]
- [包含的功能点2]

❌ 不测范围（OUT）：
- [关联但本次不测的功能，明确排除]
- [历史功能、其他模块等]

依赖与前置：
- [需要的接口文档/原型/测试账号/测试环境]
```

> 把 OUT 列出来和把 IN 列出来同样重要——评审时说不清"不测什么"会导致反复扯皮。

### Step 2 — 结构化可测项拆解

将需求拆解为独立、可验证的功能点，按优先级分组输出。

拆解原则：
- 一条可测项 = 一个可独立验证的行为（不是"整个流程"）
- 有明确的通过/失败标准
- 粒度：用户可感知的操作 + 系统响应

输出为表格：

| ID | 可测项描述 | 优先级 | 类型 |
|---|---|---|---|
| F01 | [具体行为，动宾结构] | P0 | 正向 |
| F02 | [边界条件] | P1 | 边界 |
| F03 | [异常处理] | P1 | 异常 |

优先级定义：
- **P0**：核心主流程，阻断性，缺失直接影响业务
- **P1**：重要功能，影响主要用户路径
- **P2**：边缘场景、兜底逻辑、体验细节

类型：正向 / 边界 / 异常 / 兼容 / 性能

### Step 3 — 风险识别

主动识别需求中的风险，提前暴露问题：

| 风险描述 | 风险类型 | 严重程度 | 建议措施 |
|---|---|---|---|
| [具体风险] | 业务/技术/体验 | 高/中/低 | [缓解建议] |

移动端/小程序/H5 特有风险需重点关注：
- 网络状态切换（弱网/断网/WiFi↔4G 切换）
- 前后台切换、应用被系统杀死后的状态恢复
- iOS vs Android 行为差异
- 小程序宿主环境限制（微信版本、小程序包大小）
- H5 在不同容器的差异（微信内 WebView vs 系统浏览器）

### Step 4 — 待澄清问题清单

区分受众，方便在评审会上有针对性地提问：

| # | 问题描述 | 面向 | 优先级 |
|---|---|---|---|
| Q1 | [具体疑问，可量化] | 产品 | 高 |
| Q2 | [技术实现相关] | 开发 | 中 |
| Q3 | [交互细节] | 设计 | 低 |

面向类型：产品 / 开发 / 设计

> 待澄清问题的质量决定了需求评审的效率。优先问「如果没有明确定义会直接影响测试覆盖」的问题。

### 阶段一输出文件

执行顺序：
1. **先在对话内展示**完整分析内容（Markdown，可复制）
2. 展示完毕后询问用户：「以上内容确认没问题吗？确认后我将保存为文件。」
3. 用户确认后，再保存：`需求分析_[功能名]_[YYYYMMDD].md` → 当前工作目录
4. 完成后汇报摘要：可测项数量（P0/P1/P2 各几条）、高优风险数、高优待澄清问题数

> 未经用户确认，不得主动写入任何文件。

---

## 阶段二：完整测试点生成（含思维导图 + 测试用例）

**输入**：已澄清的需求 + 阶段一产出（可选，有则复用）。

阶段二分两步独立输出：**① 测试点思维导图** + **② 测试用例表**，用户分别确认后保存。

---

### Step A — 测试点思维导图

#### 结构规则

按功能模块多层级组织，叶子节点为简洁测试点标题，**不需要前置条件、操作步骤、预期结果**。

```
[功能名称]
├── 1 [功能模块]
│   ├── 1.1 [子功能]
│   │   ├── 展示: 场景1/场景2/场景3
│   │   ├── 跳转: 成功/失败/异常兜底
│   │   ├── 异常: 无网络/弱网/超时/登录过期
│   │   └── 防抖: 重复点击/快速连续操作
│   └── 1.2 [另一子功能]
│       ├── 正常: 核心场景描述
│       └── 边界: 空数据/缺失字段/加载失败
├── 2 [功能模块]
└── ...
```

叶子节点格式：`[维度关键词]: 场景1/场景2/...`（参考维度：展示、跳转、异常、防抖、边界、兼容、生命周期、权限）

#### 覆盖维度（分布在各子功能节点下）

| 维度 | 重点检查项 |
|---|---|
| 展示 | 正常数据、空数据、缺失字段、加载失败/占位图 |
| 跳转 | 成功/失败提示、返回链路、兜底页 |
| 异常 | 无网络/弱网/超时/登录过期/接口错误码 |
| 防抖 | 重复点击、快速连续操作、重复提交 |
| 边界 | 数值上下限、字符长度极值、空/null |
| 生命周期 | 锁屏/返回键/切换 App/接听电话后状态 |
| 平台差异 | iOS vs Android；微信内/独立浏览器/App 内 H5 |
| 兼容性 | 低版本系统、主流机型、深色模式 |

#### 输出格式

在对话中以 Markdown 树形结构展示，用户确认后询问保存格式：
- `.xmind`：可用 XMind 直接打开，调用 `scripts/gen_xmind.py` 生成
- `.mm`：FreeMind 格式，更轻量，调用 `scripts/gen_mm.py` 生成

---

### Step B — 测试用例表

测试用例在测试点基础上补充完整执行信息，供执行阶段使用。

#### 测试用例字段

| 字段 | 说明 |
|---|---|
| ID | TC-功能模块编号-序号，如 TC-101 |
| 标题 | 简洁，20字内 |
| 优先级 | P0 / P1 / P2 |
| 前置条件 | 简述环境/账号/数据准备，不需要完整步骤 |
| 操作步骤 | 分步，1. xxx 2. xxx |
| 预期结果 | 可量化、明确的判断标准 |
| 所属模块 | 对应思维导图的功能模块 |

**P0**：核心主流程，阻断性 / **P1**：重要功能路径 / **P2**：边缘场景、兜底、体验细节

#### 输出格式

在对话中以 Markdown 表格展示，用户确认后询问保存格式：
- **Excel**（推荐）：`.xlsx`，每个功能模块一个 Sheet，调用 `scripts/gen_testcase_excel.py` 生成
- **Markdown**：`.md`，所有用例在一个文件内

---

### 阶段二执行顺序

1. **生成测试点**（Step A）：在对话中展示完整思维导图结构
2. 询问用户：「测试点确认没问题吗？确认后请选择保存格式（.xmind / .mm）。」
3. 用户确认后保存思维导图文件：`测试点_[功能名]_[YYYYMMDD].xmind` 或 `.mm`
4. **生成测试用例**（Step B）：在对话中展示完整用例表
5. 询问用户：「测试用例确认没问题吗？确认后请选择保存格式（Excel / Markdown）。」
6. 用户确认后保存用例文件：`测试用例_[功能名]_[YYYYMMDD].xlsx` 或 `.md`
7. 完成后汇报：测试点总数、用例总数（P0/P1/P2 各几条）、各维度覆盖情况

> 未经用户确认，不得主动写入任何文件。Step A 和 Step B 分别独立确认，不要合并。

---

## 输出规范

- 平台标注清楚（不要泛泛说"移动端"，要写具体：iOS/Android/微信小程序/H5）
- 需求未提及的交互逻辑，放入待澄清而非自行假设
- 风险严重程度「高」的必须有对应的 P0 可测项覆盖
- 思维导图：.xmind 调用 `scripts/gen_xmind.py`，.mm 调用 `scripts/gen_mm.py`，不要手写 XML
- 测试用例 Excel：调用 `scripts/gen_testcase_excel.py`，不要手写 Excel 格式
